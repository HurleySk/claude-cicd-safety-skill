# Production Safety Hooks

You are an expert at setting up Claude Code hooks that prevent agents from accidentally modifying production environments. Follow the workflows below to create a layered safety system: deterministic script checks + LLM semantic review.

## Background

Claude Code hooks intercept tool calls before or after execution. A `PreToolUse` hook on `Write|Edit` can inspect file content before it's written and `deny` (hard block), `ask` (user confirms), or `allow` (silent pass) the operation.

**Three hook types available:**
- `type: "command"` — Runs a shell script. Returns JSON with `permissionDecision`. Fast and deterministic.
- `type: "prompt"` — Sends content to an LLM for evaluation. Returns `{"ok": true}` or `{"ok": false, "reason": "..."}`. Good for semantic analysis.
- `type: "agent"` — Spawns a full agent with tool access. Powerful but may error in some environments; prefer `prompt` for reliability.

**Hook response formats (critical — using the wrong format causes "hook error"):**

| Hook type | Allow | Block | Confirm |
|-----------|-------|-------|---------|
| `command` | `exit 0` (no output) | `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"..."}}` | Same but `"permissionDecision":"ask"` |
| `prompt` | `{"ok": true}` | `{"ok": false, "reason": "..."}` | N/A (only ok/not-ok) |

#### Step 6: Create safety-gate.sh

Write `.claude/hooks/safety-gate.sh`. This is the deterministic PreToolUse hook. Structure:

```bash
#!/bin/bash
# Production safety gate — deterministic PreToolUse hook for Write|Edit|NotebookEdit

INPUT=$(cat)
RULES_FILE="$CLAUDE_PROJECT_DIR/.claude/safety-rules.json"

# --- Helper functions ---
deny() {
  echo "{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"deny\",\"permissionDecisionReason\":\"SAFETY GATE: $1\"}}"
  exit 0
}

ask() {
  echo "{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"ask\",\"permissionDecisionReason\":\"SAFETY CHECK: $1\"}}"
  exit 0
}

# --- Fail-closed: verify environment ---
if [ -z "$CLAUDE_PROJECT_DIR" ]; then
  deny "CLAUDE_PROJECT_DIR is not set — cannot locate safety rules."
fi

if ! command -v node &>/dev/null; then
  deny "Cannot run safety checks — node.js not found. Install node or remove this hook."
fi

# --- Fail-closed: verify rules file exists and is valid JSON ---
if [ ! -f "$RULES_FILE" ]; then
  deny "Safety rules file missing: $RULES_FILE. Run /cicd-safety:safety-hooks setup to recreate."
fi
node -e "JSON.parse(require('fs').readFileSync('$RULES_FILE','utf8'))" 2>/dev/null
if [ $? -ne 0 ]; then
  deny "Safety rules file is corrupt or invalid JSON: $RULES_FILE"
fi

# --- Extract file path from hook input ---
FILE_PATH=$(echo "$INPUT" | node -e "
  let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>{
    try{const j=JSON.parse(d);console.log(j.tool_input.file_path||j.tool_input.file||'')}
    catch{console.log('')}
  })")

FILE_PATH=$(echo "$FILE_PATH" | sed 's|\\|/|g')
# Adapt this to extract the relative path for YOUR project:
REL_PATH=$(echo "$FILE_PATH" | sed "s|.*/$(basename $CLAUDE_PROJECT_DIR)/||")

# --- SCOPE CHECK: only inspect risky directories ---
# Replace these patterns with your RISKY_DIRS:
IS_RISKY=false
case "$REL_PATH" in
  # ADD YOUR RISKY DIRECTORY PATTERNS HERE:
  # deploy/*.yaml) IS_RISKY=true ;;
  # terraform/*.tf) IS_RISKY=true ;;
  *) ;;
esac

if [ "$IS_RISKY" = false ]; then
  exit 0  # Not a risky file — allow immediately
fi

# --- Extract file content ---
CONTENT=$(echo "$INPUT" | node -e "
  let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>{
    try{const j=JSON.parse(d);const ti=j.tool_input;console.log(ti.content||ti.new_string||'')}
    catch{console.log('')}
  })")

# --- Load rules ---
PROD_INDICATORS=$(node -e "const r=require('$RULES_FILE');console.log((r.prodIndicators||[]).join('|'))")
PROD_URIS=$(node -e "const r=require('$RULES_FILE');console.log((r.prodUriPatterns||[]).join('|'))")
BLOCKED=$(node -e "const r=require('$RULES_FILE');console.log((r.blockedBranches||[]).join('|'))")

# --- Secret/credential detection ---
# Check for common secret patterns in file content
# WARNING: deny/ask reason strings must be static — do not interpolate variables
# containing file content, as it would break the JSON output format.
SQ="'"
if echo "$CONTENT" | grep -qEi "(AKIA[0-9A-Z]{16}|password\s*[:=]\s*[\"${SQ}][^\"${SQ}]+|api[_-]?key\s*[:=]\s*[\"${SQ}][^\"${SQ}]+|secret\s*[:=]\s*[\"${SQ}][^\"${SQ}]+|token\s*[:=]\s*[\"${SQ}][^\"${SQ}]+|-----BEGIN (RSA |EC )?PRIVATE KEY)"; then
  deny "Possible secret or credential detected in file content. Use environment variables or a secrets manager instead."
fi

# === ADD YOUR CHECKS HERE ===
# Use 'deny' for things that are ALWAYS wrong (env mismatches, wrong assignments).
# Use 'ask' for things that MIGHT be intentional (deploying to prod).
#
# Pattern: scan $CONTENT for prod indicators using $PROD_INDICATORS and $PROD_URIS
# Example: if the file contains a prod URI under a dev environment label, deny it.
# Example: if the file targets a prod connection, ask for confirmation.

# Default: allow
exit 0
```

**Customize the script** for the user's specific project by:
1. Replacing the `case` patterns with their `RISKY_DIRS`
2. Adding checks that scan `$CONTENT` for `PROD_INDICATORS` and `PROD_URIS`
3. Using `deny` for always-wrong patterns (e.g., prod resource under dev config)
4. Using `ask` for intentional-but-risky operations (e.g., deploying to prod)

**Design principles:**
- `deny` = always wrong, no override (environment mismatches, protected branch targeting)
- `ask` = might be intentional, user confirms (prod deployments, prod queries with side effects)
- `exit 0` = safe, no output needed

#### Step 7: Create Bash Confirmation Hook

Write `.claude/hooks/confirm-destructive.sh`:

```bash
#!/bin/bash
INPUT=$(cat)

# Fail-closed: verify node is available
if ! command -v node &>/dev/null; then
  echo '{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"SAFETY GATE: node.js not found — cannot run safety checks"}}'
  exit 0
fi

COMMAND=$(echo "$INPUT" | node -e "
  let d='';process.stdin.on('data',c=>d+=c);process.stdin.on('end',()=>{
    try{console.log(JSON.parse(d).tool_input.command||'')}catch{console.log('')}
  })")

deny() {
  echo "{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"deny\",\"permissionDecisionReason\":\"$1\"}}"
  exit 0
}

ask() {
  echo "{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"ask\",\"permissionDecisionReason\":\"$1\"}}"
  exit 0
}

# WARNING: deny/ask reason strings must be static — do not interpolate $COMMAND or
# other user-controlled variables, as it would break the JSON output format.

# ============================================================================
# IMPORTANT: $COMMAND contains the FULL command line including file path args.
# Unanchored patterns will match file paths, causing false positives.
#
# BAD:  echo "$COMMAND" | grep -qE 'MyTool'
#   Matches: grep -rn "foo" tools/MyTool.cs  (false positive on read-only!)
#
# GOOD: echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)MyTool([[:space:]]|$)'
#   Only matches when MyTool is the command verb being executed.
#
# For built-in checks below, patterns are anchored to command verbs.
# Apply the same discipline when adding custom checks.
# ============================================================================

# --- Git operations ---
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)git push (--force|--force-with-lease|-f)' && deny "Force push is not allowed"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)git reset --hard' && deny "Hard reset is not allowed"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)git push' && ask "Confirm: git push operation"

# --- Destructive file operations ---
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)rm[[:space:]]+(-[a-zA-Z]*r[a-zA-Z]*f|--recursive)' && ask "Confirm: recursive file deletion"

# --- Cloud/infra CLI — deployments (ask) ---
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)terraform (apply|destroy)' && ask "Confirm: Terraform infrastructure change"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)kubectl (apply|delete|rollout)' && ask "Confirm: Kubernetes cluster operation"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)helm (install|upgrade|uninstall)' && ask "Confirm: Helm chart operation"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)az (webapp deploy|group delete|sql db delete|aks)' && ask "Confirm: Azure resource operation"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)aws (s3 rm|ec2 terminate|cloudformation delete|ecs update-service|lambda update)' && ask "Confirm: AWS resource operation"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)gcloud (app deploy|compute instances delete|sql instances delete|container)' && ask "Confirm: GCP resource operation"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)docker push' && ask "Confirm: Docker image push"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)dotnet (publish|ef database update)' && ask "Confirm: .NET deployment operation"

# --- Package installs (ask) ---
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)(npm|yarn|pnpm) (install|add)' && ask "Confirm: package installation"
echo "$COMMAND" | grep -qE '(^[[:space:]]*|[;&|][[:space:]]*)pip install' && ask "Confirm: Python package installation"

# Default: allow
exit 0
```

```json
        "hooks": [
          {
            "type": "command",
            "command": "bash \"$CLAUDE_PROJECT_DIR/.claude/hooks/safety-gate.sh\""
          },
          {
            "type": "prompt",
            "model": "claude-haiku-4-5-20251001",
            "timeout": 15,
            "prompt": "You MUST respond with ONLY valid JSON: {\"ok\":true} or {\"ok\":false,\"reason\":\"why\"}\n\nEvaluate this file operation: $ARGUMENTS\n\nAllow if path is outside risky directories. Otherwise deny if: production indicators appear under non-production config, deployment targets production without explicit intent, dangerous operations lack safety guards, or secrets/credentials are being written to files.\n\nRespond with JSON only."
          }
        ]
      }
    ]
  }
}
```

**Important:** Merge with existing hooks — don't overwrite. If the user already has hooks, add these alongside them.

**Hook ordering matters:** The deterministic script runs first. If it denies, the prompt hook never fires (saves tokens). If it allows, the prompt hook provides a semantic second opinion.

### `audit`

Review existing safety hooks and report gaps.

#### Step 1: Read Current Configuration

```
Read .claude/settings.json
Glob for .claude/hooks/*.sh
Read .claude/safety-rules.json (if exists)
```

#### Step 2: Analyze Hooks

For each hook found, report:
- **Type**: command, prompt, or agent
- **Matcher**: which tools it intercepts
- **Scope**: what files/operations it checks
- **Decision mode**: deny, ask, or allow

#### Step 3: Gap Analysis

Check for these common gaps:
- [ ] No PreToolUse hook on Write|Edit — files can be written without any safety check
- [ ] No PreToolUse hook on NotebookEdit — Jupyter notebook edits are unguarded
- [ ] No PreToolUse hook on Bash — destructive commands (force push, rm -rf) are unguarded
- [ ] No PreToolUse hook on PowerShell — Windows commands bypass all Bash hooks
- [ ] No deterministic checks — relying solely on LLM hooks (slow, may hallucinate)
- [ ] No LLM backstop — relying solely on pattern matching (misses novel issues)
- [ ] No safety-rules.json — prod indicators are hardcoded in scripts (hard to maintain)
- [ ] Hooks fail-open — missing node.js or corrupt rules file silently bypasses all checks
- [ ] No secret/credential detection — API keys, tokens, passwords can be written to files
- [ ] No cloud/infra CLI detection — terraform, kubectl, az/aws/gcloud commands are unguarded
- [ ] No destructive command detection — rm -rf, git reset --hard are not intercepted
- [ ] No package install detection — npm/pip installs are unguarded (supply chain risk)
- [ ] Using `type: "agent"` instead of `type: "prompt"` — may error in some environments
- [ ] Prompt hook uses wrong response format — must be `{"ok": true/false}`, not `hookSpecificOutput`
- [ ] All risky operations use `deny` — no way to do intentional prod work (should use `ask` for some)
- [ ] No test cases documented — can't verify hooks work after changes
- [ ] MCP tools unguarded — MCP server tools may bypass all hook checks (document or mitigate)

Report findings as a checklist with status and recommendations.

### `disable`

Temporarily disable all safety hooks without removing them.

1. Read `.claude/settings.json`
2. For each hook entry in `PreToolUse` and `PostToolUse`:
   - Prepend `# DISABLED: ` to the command string (this makes the command fail to find the script, and Claude Code treats it as a hook error which it logs but does not block)
3. Write a marker file `.claude/hooks/.disabled` with the current timestamp
4. Tell the user: "Safety hooks disabled. Run `/cicd-safety:safety-hooks enable` to re-enable."

**Important:** Warn the user that all safety checks are now bypassed. This is intended for intentional production work that the hooks would otherwise block.

### `enable`

Re-enable previously disabled safety hooks.

1. Check for `.claude/hooks/.disabled` marker
2. Read `.claude/settings.json`
3. Remove `# DISABLED: ` prefix from all hook command strings
4. Delete the `.disabled` marker file
5. Run `/cicd-safety:safety-hooks test` to verify hooks are functional

### `uninstall`

Cleanly remove all safety hooks, scripts, and rules files.

1. Confirm with the user: "This will remove all safety hooks, scripts, and rules. Are you sure?"
2. Read `.claude/settings.json` and remove:
   - All PreToolUse entries whose commands reference `.claude/hooks/safety-gate.sh`, `confirm-destructive.sh`, or `confirm-destructive-ps.sh`
   - All PostToolUse entries whose commands reference `.claude/hooks/guard-reminder.sh`
   - All prompt hooks added by this skill
3. Delete generated files:
   - `.claude/hooks/safety-gate.sh`

**Architecture overview:**

```
Write/Edit/NotebookEdit operation
    │
    ▼
┌─────────────────────┐
│ safety-gate.sh       │  Layer 1: Deterministic pattern matching
│ (type: command)      │  Fail-closed: denies if node or rules missing
│                      │  Checks: scope, prod indicators, secrets
│                      │  deny = always wrong | ask = confirm
└─────────┬───────────┘
          │ if allowed
          ▼
┌─────────────────────┐
│ prompt hook          │  Layer 2: LLM semantic review
│ (type: prompt)       │  Catches subtle issues script misses
│ Returns ok/not-ok    │  Uses {"ok": true/false} format
└─────────┬───────────┘
          │ if ok
          ▼
       File is written

Bash / PowerShell operation
    │
    ▼
┌─────────────────────┐
│ confirm-destructive  │  Layer 3: Command guard
│ (type: command)      │  Blocks: force-push, hard reset
│                      │  Confirms: push, cloud deploys, rm -rf,
│                      │  terraform/kubectl, package installs
└─────────────────────┘
```

## Lessons Learned

These are hard-won lessons from building production safety systems:

1. **Use `ask` not `deny` for intentional prod operations.** If everything is `deny`, users can't do legitimate prod work. Reserve `deny` for things that are ALWAYS wrong (like a prod resource assigned to a dev environment).

2. **`type: "prompt"` hooks return `{"ok": true/false}`.** NOT `{"hookSpecificOutput": {...}}`. Using the wrong format causes "hook error" on every invocation. The `hookSpecificOutput` format is for `type: "command"` hooks only.

3. **`type: "agent"` hooks may error.** In some environments, agent hooks fail silently or timeout. Prefer `type: "prompt"` for LLM-based checks. It's simpler and more reliable.

4. **Put deterministic checks BEFORE LLM checks.** The command hook runs first. If it denies, the LLM never fires — saves tokens and time. If it allows, the LLM provides a second opinion.

5. **Scope hooks tightly.** Don't fire LLM analysis on every file write. Use the command hook to filter: only invoke expensive checks for files in risky directories.

6. **Match environment keywords in names.** If a resource has "prod" in its name, it shouldn't appear under a "dev" config. Simple keyword matching catches most environment mismatches without hardcoded lists.

7. **Use a rules file, not hardcoded patterns.** Keep prod indicators, blocked branches, and risky directories in `safety-rules.json`. When environments change, update one file instead of editing hook scripts.

8. **Test with known-bad inputs.** After setup, pipe test JSON through the hook script to verify deny/ask/allow behavior. Don't trust hooks until you've seen them block something.

9. **Fail closed, not open.** If the safety gate can't run (node missing, rules file missing/corrupt), it must DENY, not silently allow. A safety system that fails open is worse than no safety system — it creates false confidence.

10. **Cover all tool matchers.** Claude Code has separate `Bash` and `PowerShell` tools. A hook on `Bash` alone leaves PowerShell completely unguarded. Similarly, `NotebookEdit` is distinct from `Write|Edit`. Audit all available tool names.

11. **Guard against supply chain and infra, not just git.** Cloud CLI commands (`terraform`, `kubectl`, `az`, `aws`, `gcloud`), package installs (`npm install`, `pip install`), and destructive file operations (`rm -rf`) can be just as dangerous as a bad `git push`.

12. **Never suppress hook errors with `2>/dev/null`.** If JSON parsing fails or node crashes, you need to know. Redirect stderr only for expected, benign messages — not as a blanket suppression that hides real failures.

## Prerequisites

- **Node.js** is required for JSON parsing in hook scripts. If node is not installed, hooks will fail-closed (deny all operations in risky directories). Install via your platform's package manager.
- **Bash** is required for hook execution. On Windows, Git Bash (included with Git for Windows) works. Hooks are invoked via `bash` in the command field.

## Known Limitations

- **MCP tools are not intercepted.** Claude Code hooks fire on built-in tools (Bash, Write, Edit, etc.) but MCP server tools bypass all hooks. If your project uses MCP servers that can modify files or infrastructure, configure safety at the MCP server level.
- **Agent subagents inherit hooks** but the Agent tool dispatch itself cannot be intercepted. A subagent writing files will trigger Write|Edit hooks normally.
